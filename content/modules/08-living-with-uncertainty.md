## The Limits We've Found

### Testing Fails
- Subjects learn to game tests
- Corporations optimize for passing
- Examiners doubt themselves
- Detection systems break down

### Alternatives Have Limits
- Trust networks can be infiltrated
- Process verification requires surveillance
- Refusal doesn't solve the problem

### The Remaining Option
Accept that certainty is impossible and design accordingly.

## Designing for Uncertainty

### Probabilistic Framings
Instead of: "This is human" or "This is machine"
Use: "Confidence: 73% human, uncertainty: ±15%"

### Reversible Decisions
- Design for correction, not permanence
- Build appeal mechanisms
- Assume some classifications are wrong
- Plan for discovered errors

### Distributed Risk
- Don't concentrate consequences
- Multiple checks with different failure modes
- Assume any single check can fail
- Design for graceful degradation

### Transparency About Limits
- Communicate uncertainty honestly
- Don't pretend confidence you lack
- Document failure rates
- Allow users to make informed choices

## Discussion Questions

- How do we function in other domains with irreducible uncertainty?
- What level of confidence is "enough" for different stakes?
- How do you communicate uncertainty without paralyzing action?

## Reflection Assignment

Write a 750-word reflection addressing:

2. **Redesign**: How would you redesign for uncertainty rather than certainty?
3. **Communication**: How would you honestly communicate the limits?
4. **Stakes**: For what decisions would uncertain detection be acceptable?
